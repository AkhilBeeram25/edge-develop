Status: active operating plan

This document is the working process for building Akhilbeeram's proprietary YOLO-family model from the current `ULTRALYTICS_MICRO/`, `YOLO_UPDATE/`, and `src/yolo_micro/` work. The goal is not to make random architecture changes. The goal is to make small, explainable YOLO-family changes, prove each one against baselines, and then carry the winning model into Rockchip RKNN edge deployment.

## 1. Operating Contract

Every architecture change follows this lifecycle:

1. State the target problem.
2. Write the expected technical benefit.
3. Identify the exact files that will change.
4. Implement the smallest useful version.
5. Add or update focused tests.
6. Run a local smoke check.
7. Run a benchmark or training experiment when the change can affect accuracy, latency, memory, or export.
8. Record results in the repo and `.codex-memory/`.
9. Commit locally only when useful or when the user asks.
10. Push only when the user explicitly says to push.

Git identity rules:

- Repository-local commits must use `Akhilbeeram <AkhilBeeram25@users.noreply.github.com>`.
- Before a commit or push, verify `git config user.name` and `git config user.email`.
- Do not use `Codex Autosave <codex-autosave@localhost>` for new commits.
- Do not push automatically.
- If the user says "push", first verify tests, Git status, and author identity, then commit any intended local changes and run `git push origin main`.

Source ownership rules:

- Keep source code in the project style. Do not add "generated by AI" banners, novelty comments, or Codex-branded comments.
- Keep technical provenance in docs, memory, and commit messages when useful.
- Do not hide benchmark failures. A proprietary model is only valuable if its measured behavior is honest.
- Preserve secrets discipline: no tokens, credentials, private keys, passwords, or credential-bearing URLs in code, docs, logs, memory, or commits.

## 2. Quality Bar

Changes inside `ULTRALYTICS_MICRO/` must meet the same practical quality bar as Ultralytics-style code:

- Keep standard `YOLO(...)`, `.train()`, `.val()`, `.predict()`, and export entry points working.
- Add new modules through normal module exports and parser integration.
- Prefer YAML-configurable behavior over hardcoded experiments.
- Keep upstream behavior unchanged unless the active model config opts into the new behavior.
- Keep loss, assigner, validator, export, and runtime hooks isolated and testable.
- Use short names that match local conventions.
- Add docs and examples only where they help a user reproduce the change.
- Add regression tests for parser, model construction, forward pass, loss or assignment behavior, and export where relevant.

Changes inside `YOLO_UPDATE/` must meet standalone model quality:

- Configs must fully describe the experiment.
- Trainer changes must support resume, checkpoints, validation metrics, and reproducible output paths.
- Model changes must pass synthetic trainability before real training.
- Metrics must include tiny-object slices when the dataset contains 2-to-5-pixel targets.

## 3. Benchmark Ladder

No model should be called better until it beats a defined baseline on the relevant metric.

Baseline set:

- Upstream-compatible YOLO baseline: `yolov8n.pt` or a matching upstream YOLO config on the same dataset and image size.
- Small-object baseline: upstream P2-style model where available, such as `yolov8-p2.yaml` or `yolo26-p2.yaml`.
- Current proprietary baseline: `ULTRALYTICS_MICRO/ultralytics/cfg/models/v8/yolov8-micro.yaml`.
- Candidate model: the proposed proprietary modification.

Accuracy metrics:

- `mAP50-95`
- `mAP50`
- precision
- recall
- false positives per image
- size-sliced recall for 2-to-5-pixel objects
- size-sliced recall for 6-to-16-pixel objects
- class-wise AP and recall
- unknown or rare-class recall when open-vocabulary work is active

Efficiency metrics:

- parameter count
- FLOPs or MACs at the configured image size
- peak GPU memory during training
- inference latency on CPU/GPU where available
- ONNX Runtime latency where available
- RKNN latency on target Rockchip board
- RKNN model size
- NPU utilization when the toolchain reports it
- preprocessing and postprocessing time, not just model time

Export parity metrics:

- PyTorch versus ONNX output shape parity
- PyTorch versus ONNX score and box numerical tolerance
- ONNX versus RKNN output shape parity
- RKNN confidence and box drift on a fixed image set
- postprocessing equivalence after RKNN export

## 4. Development Gates

Each change must pass the smallest relevant gate before moving forward.

| Gate | Required for | Minimum proof |
| --- | --- | --- |
| G0 Design | Every non-trivial change | Written reason, files touched, expected metric impact |
| G1 Build | Every code change | Import or compile check passes |
| G2 Unit | Shared logic | Focused test for the changed behavior |
| G3 Model | Architecture changes | Model builds and forward pass succeeds |
| G4 Loss | Training changes | Synthetic batch produces nonzero sane losses |
| G5 Train | Accuracy changes | Short smoke training saves `last.pt` and `best.pt` |
| G6 Benchmark | Claimed improvement | Same dataset/settings compared against baseline |
| G7 Export | Deployment changes | ONNX export and shape parity pass |
| G8 RKNN | Rockchip release | RKNN conversion, board inference, latency, and parity pass |
| G9 Docs | User-facing changes | Reproduction commands and result summary written |

- D1. Identify one bottleneck per iteration: recall, precision, latency, memory, export, or RKNN compatibility.
- D2. Keep P1/P2 evidence preservation as the core micro-object principle.
- D3. Use `SPDConv` or another phase-preserving transition only where it has a measured benefit.
- D4. Keep `MicroC2f`, `MicroSPPF`, `MicroFPNFusion`, and `MicroDetect` independently testable.
- D5. Add channel/depth scaling through YAML, not hardcoded code branches.
- D6. Add candidate neck fusion changes behind a new YAML config.
- D7. Add candidate assignment changes behind `MicroDetect` or config switches.
- D8. Keep upstream `Detect` behavior unchanged.
- D9. Add model-summary assertions for every new module.
- D10. Add forward-shape tests for every candidate config.
- D11. Add synthetic 2-pixel object smoke training for every candidate config.
- D12. Compare candidate model parameter count and FLOPs against baseline.
- D13. Reject changes that improve synthetic smoke but fail real validation.
- D14. Reject changes that materially hurt export compatibility unless accuracy gain justifies a separate edge fallback.
- D15. Document why each accepted block belongs in a YOLO-family model.

- G7. Inspect unsupported or expensive ONNX ops.

- G10. Create RKNN conversion script that mirrors RKNN Model Zoo layout.
- G11. Create quantization calibration dataset list.
- G12. Convert FP16 RKNN first.

- G14. Run RKNN board inference on fixed images.
- G15. Compare ONNX and RKNN outputs.
